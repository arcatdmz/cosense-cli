# CLIとSkillの責任境界

CLIとSkillで同じ情報を別々に持たず、基本的にSSoTにする。
以下で役割を説明する。

## `cosense [<cmd>] --help`

mechanics（usage, 引数, 戻り値スキーマ, 例）を書く。そのコマンドを使う直前に必ず読まれる

- 書く事
  - コマンドの効果の一文説明
  - キャッチフレーズ
- 書かない事
  - 「いつ使うか」「使いこなしのコツ」

## `skills/cosense/SKILL.md`

Cosense自体のdomain知識を書く。Skillの初回呼び出し時にロードされるが、contextから揮発する可能性がある

- 書く事
  - 「こういう時はこうする」のケース分岐からの、個別手順書へリンク
  - コマンド一覧表（CLI summaryのスナップショット）
- 書かない事
  - 各コマンドの詳細な引数や戻り値スキーマ

## `skills/cosense/<手順>.md`

手順の説明書。SKILL.mdで指示されたタイミングまでロードされない

- 書く事
  - そのコマンドを効果的に使うwisdom
  - 手順上注目すべきフィールド名と、その注目理由（観察指示）
- 書かない事
  - 「いつ使うか」はSKILL.mdに任せる
  - 引数・戻り値スキーマ（フィールドの値の列挙・型・基礎定義）は `cosense <cmd> --help` に任せる

### 手順.md内でのフィールド名への言及はどこまで許されるか

手順書には「この手順では `X` フィールドに注目せよ」という観察指示を書ける。
フィールド名と注目理由（wisdom）は手順書の本質なので積極的に書く。

ただし以下はスキーマ定義であり `cosense <cmd> --help` の役目:

- フィールドが取りうる値の列挙（例: `relation` の `outgoing` / `incoming` / `bidirectional`）
- 型（string, number, boolean等）
- 基礎定義（「`exists: false` = 空ページ」のような意味付け）

判断基準: AIが `cosense <cmd> --help` を読めば分かる事は書かない。読んでも分からない「このタスクで何を見るべきか・なぜ見るべきか」だけを書く。

## SSoTの粒度

抽象的言及は重複OK。
具体的な値・コマンド・スキーマは1箇所にだけ書く。
2箇所で説明が必要な場合は、片方から他方へ参照リンクで繋ぐ。

## 知識をどこに追加するか迷ったら

以下の原則に従う

- context消費を抑える
- 安易に手順書を増やさない。想定外の複雑な作業にAIが自力対応できなくなる
- 役割が明確な小さなパーツを組み合わせて、AIが創造的に作業できるよう環境を整える
